## Problem/Context
### Auto-zone architecture
- Alloy's built-in AppendFile APPENDS (accumulates), does not replace. UpdateFile is whole-file read-modify-write.
- Neither provides replace-region semantics out of the box.
- UpdateFile internally wraps in core SourceFile, but TS components need ts.SourceFile (TSLexicalScope). This caused "Expected TSLexicalScope, but got TSPackageScope" error.
- Fix: TsReplaceFile uses readFileSync directly + wraps in ts.SourceFile, bypassing UpdateFile entirely.
- Same pattern needed for Rust: RustReplaceFile wrapping the Rust SourceFile from packages/rust.

### Stable barrel architecture (designed, not yet implemented)
- One root barrel file re-exports all public types
- Manual code imports from barrel (stable path), auto-generated types move freely underneath
- Git diff on barrel = topology changelog
- Tree shaking handles unused re-exports

## Solution/Approach
Two-mode TsReplaceFile component:
1. First emit (no existing file): builds full output with sigil markers + ManualZone scaffold
2. Re-emit (existingFile prop set): reads file, replaces between alloy-{id}-start/end sigils, preserves everything outside

For Rust: same pattern but wrapping Rust SourceFile component (packages/rust/src/components/23_SourceFile.tsx) instead of ts.SourceFile.

## Key Insights
- Alloy scope incompatibility: language SourceFile components create language-specific scopes; core SourceFile creates base scope. Any file-manipulation primitive (UpdateFile, AppendFile) that wraps core SourceFile is unusable for language-scoped components. The fix is to bypass UpdateFile and use readFileSync + language SourceFile directly.
- Auto-zone sigil format follows Alloy convention: `// alloy-{id}-start` / `// alloy-{id}-end` (any comment style)
