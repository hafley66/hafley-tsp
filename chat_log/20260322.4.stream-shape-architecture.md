# Session: stream-shape-architecture

## Goal
Design the stream abstraction layer for multi-transport endpoint generation. Extend the existing transport binding system (HTTP, WS) with stream shapes (unary, server_stream, client_stream, bidi) and a common interface that can target any protocol (gRPC, NATS, MQTT, CLI, etc.).

## Current State
Architecture is designed but not yet implemented. Deep research complete across 20 protocols/frameworks/specs. Common interface model is agreed upon. Ready to implement the stream shape axis.

## Problem/Context
The existing transport binding system handles param extraction for HTTP and WS but has no concept of stream shape. OperationInfo is HTTP-biased (has `method`, `path`). Need to generalize to support:
- Four stream shapes: unary, server_stream, client_stream, bidi
- N transports with exhaustive compile-time enforcement
- Three-level constructor model: connection → channel → message (universal across all protocols)

## Solution/Approach

### Three-level constructor model (universal across all 20 researched protocols)
1. **Connection** (auth, transport config) → `PerRequest<T>`
2. **Channel** (stream ID, init params) → init-phase params on the operation
3. **Message** (per-frame payload) → message-phase params, typed as `Stream<T>`

On HTTP all three collapse into one request. On WS/gRPC/NATS they're temporally separated.

### Stream shape inferred from TypeSpec signature
- No `Stream` in input or output → `unary`
- `Stream` in output only → `server_stream`
- `Stream` in input only → `client_stream`
- `Stream` in both → `bidi`

### Manual impl IS the common interface
No runtime trait needed. The developer-written function signature encodes the stream shape:
```rust
async fn get_room(session: UserSession, room_id: Uuid) -> Room                    // unary
fn search(session: UserSession, q: String) -> impl Stream<Item = Room>             // server_stream
async fn upload(session: UserSession, chunks: impl Stream<Item = Chunk>) -> Result // client_stream
fn chat(session: UserSession, room_id: Uuid, msgs: impl Stream<Item = In>) -> impl Stream<Item = Out> // bidi
```

### Three exhaustive matrices (satisfies Record pattern)
1. `Record<TransportId, Record<ParamSource, ExtractorFn>>` -- param extraction (EXISTS)
2. `Record<TransportId, Record<InjectableScope, ExtractorFn>>` -- DI extraction (EXISTS)
3. `Record<TransportId, Record<StreamShape, StreamWiringFn>>` -- stream I/O wiring (NEW)

### OperationInfo becomes transport-agnostic
Replace HTTP-specific fields with:
```typescript
interface OperationInfo {
  name: string;
  domainPath: string[];      // ["rooms", "{room_id}", "posts"]
  verb: "read" | "create" | "update" | "delete" | "action";
  stream: StreamShape;
  params: ClassifiedParam[];
  inputType?: Refkey;        // for client_stream / bidi
  outputType: Refkey;
}
```

### AsyncAPI parallel
Our architecture mirrors AsyncAPI's model: protocol-agnostic channel definition + protocol-specific bindings as overlays. Difference: AsyncAPI describes interfaces, we generate implementations.

## Design Decisions (locked)

### Scopes stay at two: PerRequest + Shared
- PerRequest = caller identity (HTTP middleware, WS upgrade, CLI env, gRPC metadata)
- Shared = process singleton (DbPool, config, RoomManager)
- "Stream state" (room membership, cursor position) is domain state behind Shared handles, not a framework scope

### Stream instances are globally-addressable infrastructure
- Not process-local Shared<T>
- Entries in a backing store (Redis pubsub, Postgres, etc.)
- Framework extracts stream-id per transport (HTTP header, WS connection state, CLI arg)
- Backing store choice is manual-zone code

### Per-message metadata is part of the message type
- RSocket has first-class per-message metadata slots, but we follow gRPC/protobuf approach
- If a message needs metadata, it goes in the message type fields
- Simpler, type system handles it

### No param phase axis needed on ClassifiedParam
- Originally considered `phase: "init" | "message"` on SourcedParam
- Instead: init params are regular function args, message params are the Stream<T> type
- The stream shape + type signature encode the phase distinction

## Key Insights

### Research findings (20 protocols surveyed)
- Seven irreducible primitives: Connection, Channel, Message, Interaction pattern, Backpressure, Resumption, Binding
- RSocket has the strongest wire-level abstraction (4 named interaction types, per-message metadata, transport-pluggable, REQUEST_N backpressure)
- AsyncAPI has the strongest spec-level abstraction (protocol bindings as overlays)
- Every protocol separates init into connection → channel → message scopes
- The 2x2 stream shape matrix (1-or-N input x 1-or-N output) appears independently in gRPC, RSocket, and protobuf

### Framework-level findings
- tRPC's "links" are a clean transport abstraction (splitLink routes procedure types to different transports)
- Phoenix Channels: WS with long-poll fallback, transparent to application code, CRDT-based presence tracking
- Connect RPC: one server speaks gRPC + gRPC-Web + Connect protocol simultaneously (cleanest multi-protocol tunneling)
- Temporal/Restate: durable execution treats workflow as event journal (replay model)
